The package exposes a thin command-line adapter over the default project
harness runner:

```shell
python-project-harness [--json] [--no-tests] [--source-dir DIR] [--test-dir DIR] [--extra-path PATH] [PROJECT_ROOT]
python -m python_lang_project_harness [--json] [--no-tests] [--source-dir DIR] [--test-dir DIR] [--extra-path PATH] [PROJECT_ROOT]
```

When `PROJECT_ROOT` is omitted, the current working directory is used.

## Scope Options

The CLI mirrors the project runner's scan-shape options:

```shell
python-project-harness --source-dir lib --test-dir checks --extra-path tools .
python-project-harness --no-tests .
```

`--source-dir`, `--test-dir`, and `--extra-path` can be repeated. Supplying one
or more source or test roots replaces the default root names for that run.
`--extra-path` adds opt-in project paths such as `examples/`, `scripts/`,
`tools/`, or a single Python file. `--no-tests` skips test parser discovery but
keeps tests-root layout policy active.

## Output Modes

Compact text is the default output for humans and repair-oriented agents:

```shell
python-project-harness .
```

Use `--json` when a tool needs the structured `PythonHarnessReport` payload:

```shell
python-project-harness --json .
```

## Exit Codes

- `0`: no configured-blocking findings
- `1`: configured-blocking findings exist
- `2`: CLI argument or project-root error

`PY-AGENT-*` findings remain advisory by default. Advice is rendered in compact
text, but it does not change the exit code unless the caller promotes `Info`
severity through library APIs.

## Library Boundary

`run_cli_from_env()` is the console-script entrypoint. `run_cli(...)` is exposed
for tests and embeddings that want to provide explicit streams or a current
working directory. Both functions delegate to `run_python_project_harness()` and
the public renderers.
